## Zettel

**Главная мысль:** процесс в Linux - это исполняемая программа вместе с ресурсами и контекстом; новые процессы почти всегда рождаются от существующих через `fork`/`exec` или более общий `clone`, поэтому процессы образуют дерево.

## Развернутый ответ

**Процесс** - совокупность исполняемых команд, связанных ресурсов и контекста исполнения, находящаяся под управлением ОС.

В Linux процесс представлен структурой ядра `task_struct`. Она содержит множество полей, среди них:

- PID - идентификатор процесса;
- TGID - идентификатор группы потоков;
- указатель на родительский процесс;
- имя исполняемого файла;
- UID и GID пользователя;
- состояние процесса;
- указатель на таблицы страниц памяти;
- рабочая директория;
- таблица открытых файловых дескрипторов;
- контекст процессора: регистры, стек, счетчик инструкций.

Процесс - не просто программа на диске. Программа - это файл с кодом, а процесс - запущенный экземпляр с памятью, правами, дескрипторами и состоянием.

## Дерево процессов

В Linux процессы порождаются другими процессами и образуют дерево. У каждого процесса, кроме корневого, есть родитель.

Корнем пользовательского дерева является первый процесс:

- исторически `init`;
- в современных дистрибутивах чаще `systemd`.

Причины древовидной модели:

- процесс имеет одного создателя;
- родитель отвечает за чтение кода завершения ребенка;
- удобно распространять сигналы группам процессов;
- удобно управлять службами и сессиями.

Посмотреть дерево:

```bash
pstree -p
ps -ef
```

## Рождение процесса через fork

`fork()` создает копию текущего процесса.

После `fork` существуют два процесса:

- родитель;
- ребенок.

Ребенок получает:

- копию адресного пространства;
- копии файловых дескрипторов;
- обработчики сигналов;
- рабочую директорию;
- права доступа;
- окружение.

Но ребенок получает новый PID.

Возвращаемое значение `fork()`:

- `-1` - ошибка;
- `0` - код выполняется в дочернем процессе;
- PID ребенка - код выполняется в родительском процессе.

Пример:

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/types.h>

int main(void) {
    pid_t pid = fork();

    if (pid == -1) {
        perror("fork");
        return 1;
    }

    if (pid == 0) {
        printf("child: pid=%d\n", getpid());
    } else {
        printf("parent: child pid=%d\n", pid);
    }

    return 0;
}
```

## Copy-on-Write

Полное копирование памяти при каждом `fork()` было бы дорогим. Поэтому Linux использует Copy-on-Write.

Смысл:

1. После `fork()` родитель и ребенок сначала указывают на одни и те же страницы памяти.
2. Эти страницы помечаются как доступные только для чтения.
3. Если один процесс пытается изменить страницу, возникает page fault.
4. Ядро копирует страницу, и изменение получает только этот процесс.

Это делает `fork()` эффективным, особенно когда сразу после него вызывается `exec`.

## exec: замена программы процесса

Семейство `exec` не создает новый процесс. Оно заменяет текущий образ процесса другой программой.

После успешного `exec`:

- PID сохраняется;
- код и данные процесса заменяются;
- начинается выполнение новой программы;
- строки после `exec` не выполняются, если ошибки нет.

Основные варианты:

- `execl`;
- `execlp`;
- `execle`;
- `execv`;
- `execvp`;
- `execvpe`;
- `execve`.

Буквы:

- `l` - аргументы списком;
- `v` - аргументы массивом;
- `e` - явно передается окружение;
- `p` - поиск по `PATH`.

Пример:

```c
#include <unistd.h>
#include <stdio.h>

int main(void) {
    char *argv[] = {"ls", "-l", "/", NULL};
    execvp("ls", argv);
    perror("execvp");
    return 1;
}
```

## Типичный шаблон fork + exec

Большинство запусков команд в shell устроены так:

1. shell вызывает `fork()`;
2. дочерний процесс настраивает ввод-вывод, права, окружение;
3. дочерний процесс вызывает `exec()`;
4. родительский shell ждет завершения или продолжает работу, если команда фоновая.

Пример:

```c
pid_t pid = fork();

if (pid == 0) {
    execlp("date", "date", NULL);
    _exit(127);
}

waitpid(pid, NULL, 0);
```

## clone

`clone()` - более гибкий системный вызов. Он позволяет указать, какие ресурсы разделять между родителем и ребенком.

Возможные флаги:

- `CLONE_VM` - общее адресное пространство;
- `CLONE_SIGHAND` - общая таблица обработчиков сигналов;
- `CLONE_THREAD` - тот же thread group;
- `CLONE_PARENT` - родителем станет родитель текущего процесса;
- `CLONE_STOPPED` - ребенок создается остановленным.

`clone()` важен для реализации потоков и контейнерных механизмов, потому что он дает тонкий контроль над разделяемыми ресурсами.

## Состояния процесса

Процесс может:

- выполняться на CPU;
- быть готовым к выполнению;
- ждать ввода-вывода;
- быть остановленным;
- завершиться и стать зомби до чтения кода выхода.

Состояние видно через:

```bash
ps -o pid,ppid,state,cmd
cat /proc/<PID>/status
```

Типичные обозначения в `ps`:

- `R` - running/runnable;
- `S` - interruptible sleep;
- `D` - uninterruptible sleep;
- `T` - stopped;
- `Z` - zombie.

## Завершение процесса

Процесс завершает выполнение через `_exit(status_code)` или возврат из `main`, который в итоге приводит к завершению.

После завершения ребенок становится зомби, пока родитель не прочитает код выхода.

Родитель читает статус через:

- `wait`;
- `waitpid`;
- `waitid`;
- обработчик `SIGCHLD`.

Если родитель не читает статус, зомби остается в таблице процессов. Если родитель умер, ребенка обычно усыновляет `init`/`systemd`, который считывает его статус.

## Что сказать на экзамене коротко

Процесс - исполняемая программа с ресурсами и контекстом, представленная в ядре `task_struct`. Процессы образуют дерево, потому что новые процессы создаются существующими. `fork()` копирует процесс с оптимизацией Copy-on-Write, `exec()` заменяет программу внутри процесса, а `clone()` позволяет управлять разделением ресурсов и лежит в основе потоков. Завершенный ребенок становится зомби, пока родитель не считает его код выхода через `wait`.

## Code

```bash
# дерево процессов
pstree -p

# список процессов с родителями и состояниями
ps -eo pid,ppid,state,user,cmd

# информация ядра о процессе
cat /proc/<PID>/status
ls -l /proc/<PID>/fd

# запустить процесс в фоне
sleep 100 &

# дождаться фонового процесса в shell
wait %1
```
