## Zettel

**Главная мысль:** монтирование связывает конкретную файловую систему на устройстве, в памяти или по сети с каталогом единого дерева `/`; без монтирования раздел существует, но его содержимое не включено в пользовательскую иерархию.

## Развернутый ответ

В GNU/Linux нет отдельных корней `C:`, `D:`. Есть единое дерево каталогов, начинающееся с `/`. Чтобы использовать файловую систему, ее нужно **смонтировать** в точку монтирования - существующий каталог.

Пример:

```text
/dev/sdb1  ->  /mnt/data
```

После монтирования содержимое файловой системы на `/dev/sdb1` становится доступным через `/mnt/data`.

## Что участвует в монтировании

Нужны четыре ключевые вещи:

- источник: устройство, UUID, сетевой ресурс или псевдо-ФС;
- тип файловой системы: `ext4`, `xfs`, `vfat`, `tmpfs`, `proc`, `nfs` и т.д.;
- точка монтирования: каталог;
- опции монтирования.

Примеры источников:

```text
/dev/sdb1
UUID=90811ff5-ecb7-4e78-8406-1be8785fe758
LABEL=data
server:/export/share
tmpfs
```

## Точки монтирования

Точка монтирования - обычный каталог. Если в каталоге уже были файлы, после монтирования они не удаляются, но временно скрываются содержимым смонтированной ФС. После размонтирования старое содержимое снова видно.

Типичные каталоги:

- `/` - корневая файловая система;
- `/boot` - ядро, initramfs, файлы загрузчика;
- `/boot/efi` или `/boot/EFI` - EFI System Partition;
- `/home` - пользовательские данные;
- `/var` - изменяемые данные;
- `/mnt` - временное монтирование администратором;
- `/media` - съемные носители;
- `/proc`, `/sys`, `/dev` - виртуальные и служебные ФС.

## Ручное монтирование

Базовая команда:

```bash
sudo mount -t ext4 /dev/sdb1 /mnt/data
```

Если тип ФС определяется автоматически:

```bash
sudo mount /dev/sdb1 /mnt/data
```

Создать точку монтирования:

```bash
sudo mkdir -p /mnt/data
```

Посмотреть смонтированные ФС:

```bash
findmnt
mount
cat /proc/mounts
```

Размонтировать:

```bash
sudo umount /mnt/data
```

Если устройство занято, нужно найти процессы:

```bash
lsof +f -- /mnt/data
fuser -vm /mnt/data
```

## Опции монтирования

Опции задают поведение ФС:

| Опция | Смысл |
|---|---|
| `ro` | только чтение |
| `rw` | чтение и запись |
| `noexec` | запрет запуска исполняемых файлов |
| `nosuid` | игнорировать SUID/SGID |
| `nodev` | не интерпретировать файлы устройств |
| `relatime` | обновлять atime оптимизированно |
| `noatime` | не обновлять atime |
| `defaults` | набор опций по умолчанию |

Пример для временного небезопасного каталога:

```bash
sudo mount -o rw,nosuid,nodev,noexec /dev/sdb1 /mnt/data
```

## Автоматическое монтирование через `/etc/fstab`

`/etc/fstab` описывает файловые системы, которые система должна монтировать автоматически или по команде `mount`.

Формат строки:

```text
<source> <mountpoint> <type> <options> <dump> <pass>
```

Пример:

```fstab
UUID=1111-2222 /boot/efi vfat umask=0077 0 1
UUID=aaaa-bbbb / ext4 defaults,errors=remount-ro 0 1
UUID=cccc-dddd /home ext4 defaults 0 2
/swapfile none swap sw 0 0
tmpfs /tmp tmpfs defaults,nosuid,nodev 0 0
```

Поля:

- `source` - устройство, UUID, LABEL или ресурс;
- `mountpoint` - точка монтирования;
- `type` - тип ФС;
- `options` - опции;
- `dump` - использовать ли старую утилиту `dump`, обычно `0`;
- `pass` - порядок проверки `fsck`: `1` для `/`, `2` для остальных, `0` не проверять.

Почему лучше UUID:

- имя `/dev/sdb1` может измениться при другом порядке обнаружения дисков;
- UUID принадлежит самой файловой системе и стабильнее.

Проверить `fstab` без перезагрузки:

```bash
sudo mount -a
findmnt --verify
```

## Автоматическое монтирование в systemd

Современные дистрибутивы используют `systemd`. Он читает `/etc/fstab` и генерирует `.mount` units. Также можно создавать unit-файлы вручную.

Пример unit:

```ini
[Unit]
Description=Data partition

[Mount]
What=/dev/disk/by-uuid/cccc-dddd
Where=/srv/data
Type=ext4
Options=defaults

[Install]
WantedBy=multi-user.target
```

Имя unit должно соответствовать пути: `/srv/data` превращается в `srv-data.mount`.

Команды:

```bash
sudo systemctl daemon-reload
sudo systemctl enable srv-data.mount
sudo systemctl start srv-data.mount
systemctl status srv-data.mount
```

Есть также automount units, которые монтируют ФС при первом обращении:

```ini
[Automount]
Where=/srv/data
```

## Монтирование при загрузке системы

Во время загрузки ядро сначала не всегда может сразу смонтировать настоящий корень `/`, потому что могут быть нужны драйверы диска, RAID, LVM или файловой системы. Для этого используется `initramfs`.

Упрощенно:

1. Загрузчик передает ядру параметры, включая `root=...`.
2. Ядро запускает временную среду `initramfs`.
3. `initramfs` подгружает нужные модули и находит корневой раздел.
4. Настоящий корень монтируется.
5. Выполняется переключение корня.
6. Управление передается `init`/`systemd`.
7. `systemd` монтирует остальные ФС из `fstab` и unit-файлов.

## Что сказать на экзамене коротко

Монтирование - подключение файловой системы к каталогу единого дерева `/`. Ручное монтирование выполняется командой `mount`, размонтирование - `umount`, состояние видно через `findmnt` и `/proc/mounts`. Автоматическое монтирование задается в `/etc/fstab` или через `systemd` `.mount` units. Для надежности в `fstab` обычно используют UUID, а не нестабильные имена `/dev/sdX`.

## Code

```bash
# найти устройства и UUID
lsblk -f
blkid

# ручное монтирование
sudo mkdir -p /mnt/data
sudo mount -t ext4 /dev/sdb1 /mnt/data

# посмотреть результат
findmnt /mnt/data
cat /proc/mounts

# размонтировать
sudo umount /mnt/data

# проверить fstab
sudo mount -a
findmnt --verify
```
