## Zettel

**Главная мысль:** ext3/ext4 управляют диском через набор метаданных: суперблок описывает всю файловую систему, группы блоков делят пространство на локальные зоны, битовые карты показывают занятость, а inode описывает отдельный файл.

## Развернутый ответ

ext3 и ext4 относятся к файловым системам с индексными дескрипторами. Их задача - организовать блочное устройство так, чтобы можно было эффективно находить свободное место, хранить файлы, проверять целостность и восстанавливаться после сбоев.

Упрощенная структура ext4:

```text
суперблок
таблица дескрипторов групп
группа блоков 0
группа блоков 1
...
группа блоков N
```

Внутри группы обычно есть:

```text
копии служебных структур
битовая карта блоков
битовая карта inode
таблица inode
блоки данных
```

## Суперблок

**Суперблок** - главная структура, описывающая файловую систему в целом. Он находится в начале блочного устройства, а резервные копии могут храниться в некоторых группах блоков.

Суперблок отвечает на вопросы:

- сколько всего блоков;
- сколько всего inode;
- сколько свободных блоков и inode;
- какого размера блок;
- сколько блоков и inode в группе;
- когда ФС была смонтирована и записана;
- в каком состоянии ФС;
- какой UUID и имя тома;
- какая версия и какие возможности поддерживаются.

Ключевые поля из лекции:

| Поле | Назначение |
|---|---|
| `s_inodes_count` | общее число inode |
| `s_blocks_count_lo` | общее число блоков |
| `s_free_blocks_count_lo` | число свободных блоков |
| `s_free_inodes_count` | число свободных inode |
| `s_first_data_block` | первый блок данных |
| `s_log_block_size` | размер блока через логарифм |
| `s_blocks_per_group` | блоков в группе |
| `s_inodes_per_group` | inode в группе |
| `s_mtime` | время последнего монтирования |
| `s_wtime` | время последней записи |
| `s_mnt_count` | счетчик монтирований после проверки |
| `s_max_mnt_count` | предел монтирований до проверки |
| `s_magic` | магическое число ext, для ext4 `0xEF53` |
| `s_state` | состояние файловой системы |
| `s_errors` | поведение при ошибках |
| `s_uuid` | UUID файловой системы |
| `s_volume_name` | имя тома |

Практическая роль суперблока: если он поврежден, ФС может стать недоступной. Поэтому резервные копии суперблока важны для восстановления.

## Группа блоков

Группы блоков появились для локальности данных. Если хранить блоки большого или связанного набора файлов рядом, меньше перемещается головка HDD, а на SSD и современных носителях уменьшается хаотичность доступа.

**Группа блоков** содержит:

- битовую карту блоков;
- битовую карту inode;
- таблицу inode;
- блоки данных;
- иногда резервные копии суперблока и таблицы дескрипторов групп.

### Дескриптор группы блоков

Таблица дескрипторов групп хранит сведения о каждой группе. В дескрипторе есть:

| Поле | Назначение |
|---|---|
| `bg_block_bitmap_lo` | блок, где лежит bitmap блоков |
| `bg_inode_bitmap_lo` | блок, где лежит bitmap inode |
| `bg_inode_table_lo` | первый блок таблицы inode |
| `bg_free_blocks_count_lo` | свободные блоки в группе |
| `bg_free_inodes_count_lo` | свободные inode в группе |
| `bg_used_dirs_count_lo` | число каталогов в группе |
| `bg_flags` | флаги состояния группы |
| `bg_block_bitmap_csum_lo` | контрольная сумма bitmap блоков |
| `bg_inode_bitmap_csum_lo` | контрольная сумма bitmap inode |
| `bg_itable_unused_lo` | число неинициализированных inode |
| `bg_checksum` | контрольная сумма дескриптора |

## Битовые карты

Битовая карта блоков показывает, какие блоки заняты:

```text
1 - блок занят
0 - блок свободен
```

Битовая карта inode аналогично показывает занятость inode:

```text
1 - inode занят
0 - inode свободен
```

Это быстрый способ искать свободные блоки и inode без полного сканирования всей файловой системы.

## Таблица inode

Таблица inode - массив индексных дескрипторов. Номер inode является индексом в этой таблице. Каталоги хранят соответствия:

```text
имя файла -> номер inode
```

После этого ФС по номеру inode находит метаданные и блоки данных.

В ext есть специальные inode:

| Inode | Назначение |
|---|---|
| `0` | несуществующий inode |
| `1` | список бракованных блоков |
| `2` | корневой каталог `/` |
| `3` | пользовательская квота |
| `4` | групповая квота |
| `5` | загрузчик |
| `6` | каталог восстановления |
| `7` | зарезервированная группа дескрипторов |
| `8` | журнал |
| `11` | первый обычно незарезервированный inode, часто `/lost+found` |

## Индексный дескриптор

**Inode** описывает файл как объект файловой системы. Он обычно хранит:

- тип файла;
- права доступа;
- UID владельца;
- GID группы;
- размер файла;
- временные метки;
- счетчик жестких ссылок;
- ссылки на блоки данных или экстенты;
- флаги;
- информацию для ACL и расширенных атрибутов, если они используются.

Имя файла inode не хранит. Поэтому переименование файла обычно меняет запись каталога, а не inode.

В ext4 inode обычно имеет размер 256 байт. В старых ext-системах размер мог быть меньше.

## Данные файла в ext3 и ext4

В ext3 исторически использовалась блочная адресация: inode указывал на блоки данных через прямые и косвенные ссылки. Это работает, но плохо масштабируется для больших файлов и может давать фрагментацию.

В ext4 используются **экстенты**. Экстент описывает непрерывный диапазон блоков. Это компактнее и быстрее:

```text
логический диапазон файла -> физический диапазон блоков
```

Преимущества:

- меньше метаданных для больших файлов;
- быстрее последовательное чтение;
- ниже фрагментация;
- проще предварительно выделять место.

## Журнал в ext3/ext4

В ext3 появился журнал. В ext4 журналирование сохранено и улучшено.

Журнал нужен, чтобы после сбоя восстановить согласованность метаданных. Для ext3 в лекции указаны режимы:

- `journal` - журналируются данные и метаданные;
- `ordered` - журналируются метаданные, данные записываются до фиксации метаданных;
- `writeback` - метаданные журналируются, но порядок записи данных слабее.

Служебный inode `8` связан с журналом.

## Что происходит при создании файла

Упрощенный алгоритм:

1. Файловая система ищет свободный inode по inode bitmap.
2. Помечает inode занятым.
3. Заполняет inode: тип, права, владелец, группа, времена.
4. Добавляет запись в каталог: имя -> inode.
5. При записи данных ищет свободные блоки по block bitmap.
6. Помечает блоки занятыми.
7. Записывает данные и обновляет inode.
8. Для ext3/ext4 важные изменения проходят через журнал.

## Что сказать на экзамене коротко

В ext3/ext4 суперблок описывает всю файловую систему, группы блоков делят пространство на локальные области, дескрипторы групп указывают на bitmap и таблицы inode, bitmap показывают занятость блоков и inode, а inode хранит метаданные файла и ссылки на данные. ext3 добавляет журнал, ext4 улучшает структуру за счет экстентов, контрольных сумм и масштабируемости.

## Code

```bash
# показать inode файла
ls -li file.txt

# подробные метаданные файла
stat file.txt

# параметры суперблока ext
sudo tune2fs -l /dev/sdb1

# найти резервные суперблоки
sudo mke2fs -n /dev/sdb1

# отладочный просмотр ext-ФС
sudo debugfs /dev/sdb1
```
