## Zettel

**Главная мысль:** GNU/Linux отличается от многих пользовательски-ориентированных ОС тем, что администратор работает с открытой, модульной, файлово-центричной и репозиторно управляемой системой, где почти все ресурсы представлены единым интерфейсом, а политика доступа и конфигурация выражены явно.

## Развернутый ответ

**Операционная система** - базовое системное ПО, которое управляет вычислительным узлом и дает универсальный интерфейс между аппаратным обеспечением, программами и пользователем. В администрировании это означает, что главная задача администратора - не просто запускать приложения, а управлять ресурсами, политиками, безопасностью, загрузкой, файловыми системами, службами и обновлениями.

**GNU/Linux** - семейство Unix-подобных ОС на базе ядра Linux, утилит GNU и других системных компонентов. Linux распространяется не как один монолитный продукт, а как набор дистрибутивов: Debian, Ubuntu, Fedora, Arch Linux, RHEL и другие. Дистрибутив включает ядро, системные утилиты, пакетный менеджер, установщик, конфигурационные соглашения и часто графическую среду.

### 1. Открытость и свободная лицензия

Linux относится к свободному ПО и распространяется под GNU GPL 2.0. Это дает пользователю четыре ключевые свободы:

- запускать программу в любых целях;
- изучать работу программы и адаптировать ее под себя;
- распространять копии;
- улучшать программу и публиковать изменения.

Для администратора это важно практически:

- можно проверять исходный код и патчи;
- можно собирать ядро и пакеты с нужными опциями;
- можно отлаживать поведение системы глубже, чем в закрытых ОС;
- поддержка может идти не только от производителя, но и от сообщества или сторонних компаний.

В проприетарных ОС администратор чаще ограничен документацией и официальными средствами производителя. В GNU/Linux администратор может дойти до исходников ядра, системных утилит, сервисов и конфигурационных файлов.

### 2. Дистрибутивная модель

Linux не равен одному конкретному продукту. Есть ядро Linux, а есть дистрибутивы, которые собирают систему в готовый комплект. Это влияет на администрирование:

- команды и принципы часто общие, но детали зависят от дистрибутива;
- пакетные менеджеры различаются: `apt`/`dpkg` в Debian/Ubuntu, `dnf`/`rpm` в Fedora/RHEL, `pacman` в Arch;
- структура конфигурации обычно похожа, но имена пакетов, сервисов и пути могут отличаться;
- поддержка и жизненный цикл зависят от политики дистрибутива.

Например, сервер на Debian администрируется иначе, чем rolling-release система Arch Linux: в Debian приоритетом обычно являются стабильность и предсказуемые обновления, а в Arch - свежие версии пакетов и ручной контроль.

### 3. Единая иерархия файловой системы

В Linux вся файловая система начинается с одного корня `/`. У Windows обычно несколько корней: `C:`, `D:` и т.д. В Linux диски, разделы, сетевые файловые системы и виртуальные файловые системы подключаются в единую иерархию через точки монтирования.

Административное следствие:

- системные данные имеют ожидаемые места по FHS;
- `/etc` содержит конфигурацию;
- `/var` содержит изменяемые данные, логи, кэш, очереди;
- `/home` содержит пользовательские каталоги;
- `/dev` представляет устройства;
- `/proc` и `/sys` дают доступ к данным ядра и процессов;
- `/boot` содержит ядро, initramfs и файлы загрузчика.

Такой подход делает систему удобной для скриптов, резервного копирования, диагностики и разграничения разделов.

### 4. Принцип "все есть файл"

В Unix-подобных системах многие объекты представлены как файлы или файловые дескрипторы:

- обычные файлы;
- каталоги;
- устройства в `/dev`;
- каналы;
- сокеты;
- псевдофайлы в `/proc` и `/sys`;
- стандартные потоки процесса: `stdin`, `stdout`, `stderr`.

Для администратора это означает единый способ работы: открыть, прочитать, записать, закрыть. Например, можно читать информацию о процессоре из `/proc/cpuinfo`, список монтирований из `/proc/mounts`, а вывод команды перенаправлять в файл или в `/dev/null`.

### 5. Явная модель прав доступа

Классическая модель Linux строится вокруг трех категорий:

- владелец файла;
- группа файла;
- остальные пользователи.

Для каждой категории задаются права `r`, `w`, `x`. Дополнительно существуют специальные биты SUID, SGID и Sticky Bit. Такая модель проще для базового понимания, чем сложные ACL в некоторых ОС, и хорошо подходит для командной строки и автоматизации.

Администратор управляет правами командами:

```bash
chmod 750 script.sh
chown root:admins script.sh
chgrp developers project/
```

### 6. Централизованная установка и обновление ПО

В GNU/Linux ПО обычно устанавливается из репозиториев через пакетный менеджер. Это отличается от Windows-подхода, где часто скачивают отдельный установщик для каждого приложения.

Плюсы для администратора:

- единая база установленных пакетов;
- автоматическое разрешение зависимостей;
- централизованные обновления безопасности;
- проверка подписи пакетов;
- простое удаление и аудит.

Пример:

```bash
sudo apt update
sudo apt install nginx
sudo apt upgrade
```

### 7. Текстовая конфигурация и автоматизация

Linux исторически ориентирован на командную строку, текстовые конфиги и автоматизацию. Это важно для серверов:

- конфиги можно хранить в Git;
- изменения можно применять скриптами;
- легко автоматизировать массовое администрирование;
- диагностика доступна без графической среды.

Примеры административных файлов:

```text
/etc/passwd
/etc/group
/etc/fstab
/etc/ssh/sshd_config
/etc/systemd/system/*.service
```

### 8. Службы и фоновые процессы

Linux-сервер обычно работает как набор демонов: `sshd`, `cron`, `nginx`, `systemd-journald`, `NetworkManager` и т.д. Современные дистрибутивы чаще всего используют `systemd`, который управляет службами, монтированиями, журналами и целевыми состояниями системы.

Базовые команды:

```bash
systemctl status sshd
systemctl restart nginx
systemctl enable postgresql
journalctl -u nginx
```

## Что сказать на экзамене коротко

GNU/Linux отличается открытостью, дистрибутивной моделью, единой иерархией файловой системы, принципом "все есть файл", явной моделью прав доступа, централизованным пакетным управлением и высокой пригодностью к автоматизации. Для администратора это означает больше контроля и прозрачности, но также больше ответственности: нужно понимать файловую систему, права, службы, процессы, загрузку и пакетную базу.

## Code

```bash
# определить дистрибутив
cat /etc/os-release

# посмотреть версию ядра
uname -a

# посмотреть корневую иерархию
ls /

# узнать смонтированные файловые системы
findmnt

# посмотреть процессы
ps aux

# посмотреть системный журнал
journalctl -b
```
