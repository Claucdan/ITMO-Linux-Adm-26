## Развернутый ответ

## Демоны

**Демон** - фоновый процесс, который работает непрерывно и независимо от пользовательской сессии. Он ожидает события или запросы и обрабатывает их без прямого участия пользователя.

Примеры демонов:

- `sshd` - удаленный вход по SSH;
- `cron` - запуск задач по расписанию;
- `nginx` - веб-сервер;
- `systemd-journald` - журналирование;
- `NetworkManager` - управление сетью;
- `[kthreadd]` - поток ядра с PID 2, корень поддерева потоков ядра.

Особенности демона:

- нет управляющего терминала;
- стандартные потоки обычно закрыты или перенаправлены в `/dev/null`;
- логи пишутся в системный журнал или отдельный файл;
- процесс часто записывает PID в pid-файл;
- демон должен корректно реагировать на сигналы;
- обычно запускается от отдельного пользователя с минимальными правами.

## Запуск фонового процесса из shell

Команда с `&` запускается в фоне:

```bash
python3 script.py &
```

Но такой процесс все еще связан с текущей shell-сессией. При закрытии терминала он может получить `SIGHUP` и завершиться.

Чтобы отвязать:

```bash
python3 script.py &
disown %1
```

Или:

```bash
nohup python3 script.py &
```

Управление задачами:

```bash
jobs
fg %1
bg %1
```

`Ctrl+Z` останавливает текущую foreground-задачу, после чего `bg` может продолжить ее в фоне.

## Классическая демонизация в C

Классический алгоритм создания демона:

1. Вызвать `fork()`.
2. Завершить родительский процесс через `_exit()`.
3. В дочернем процессе вызвать `setsid()`, чтобы создать новую сессию и отвязаться от терминала.
4. Переопределить обработчик `SIGHUP`.
5. Сделать второй `fork()`, чтобы процесс не смог снова стать лидером сессии с управляющим терминалом.
6. Установить `umask()`.
7. Перейти в корень через `chdir("/")`.
8. Закрыть унаследованные файловые дескрипторы.
9. Перенаправить `stdin`, `stdout`, `stderr` в `/dev/null`.
10. Начать основную работу и писать логи.

Упрощенный фрагмент:

```c
pid_t pid = fork();
if (pid > 0) {
    _exit(0);
}

setsid();
chdir("/");
umask(0);

freopen("/dev/null", "r", stdin);
freopen("/dev/null", "w", stdout);
freopen("/dev/null", "w", stderr);
```

## Демоны под systemd

Современный способ - не демонизировать программу вручную, а запускать ее как службу под управлением `systemd`.

Пример:

```ini
[Unit]
Description=My Application
After=network.target

[Service]
Type=simple
User=myappuser
ExecStart=/usr/bin/myapp
Restart=on-failure
StandardOutput=journal

[Install]
WantedBy=multi-user.target
```

Команды:

```bash
sudo systemctl start myapp
sudo systemctl stop myapp
sudo systemctl restart myapp
sudo systemctl enable myapp
systemctl status myapp
journalctl -u myapp
```

`systemd` берет на себя:

- запуск;
- перезапуск при сбое;
- логирование;
- зависимости;
- ограничение прав и ресурсов;
- включение в загрузку системы.

## Сигналы

**Сигнал** - сообщение процессу от операционной системы или другого процесса. Для процесса сигнал похож на асинхронное прерывание.

Сигналы нужны, потому что процессы изолированы: у них разные адресные пространства и они не могут напрямую управлять друг другом без механизмов ОС.

Источники сигналов:

- ядро;
- другой процесс;
- терминал;
- сам процесс;
- события дочерних процессов.

Послать сигнал можно командой:

```bash
kill -TERM <PID>
kill -9 <PID>
pkill nginx
```

Несмотря на название, `kill` не обязательно убивает процесс. Она отправляет сигнал.

## Стандартные сигналы

| Сигнал | Код | Назначение | Типичное действие |
|---|---:|---|---|
| `SIGHUP` | 1 | обрыв терминала, часто перечитать конфиг | завершение или reload |
| `SIGINT` | 2 | прерывание с клавиатуры `Ctrl+C` | завершение |
| `SIGQUIT` | 3 | выход с дампом `Ctrl+\` | завершение + core dump |
| `SIGILL` | 4 | недопустимая инструкция | завершение + core dump |
| `SIGFPE` | 8 | арифметическая ошибка | завершение + core dump |
| `SIGKILL` | 9 | принудительное убийство | немедленное завершение |
| `SIGUSR1` | 10 | пользовательский сигнал | зависит от программы |
| `SIGSEGV` | 11 | ошибка доступа к памяти | завершение + core dump |
| `SIGUSR2` | 12 | пользовательский сигнал | зависит от программы |
| `SIGPIPE` | 13 | запись в канал без читателя | завершение |
| `SIGTERM` | 15 | вежливая просьба завершиться | завершение |
| `SIGCHLD` | 17 | событие дочернего процесса | обычно игнорируется или обрабатывается |
| `SIGCONT` | 18 | продолжить выполнение | продолжение |
| `SIGSTOP` | 19 | остановить процесс | остановка |
| `SIGTSTP` | 20 | остановка с терминала `Ctrl+Z` | остановка |

Коды указаны для типичных Linux-архитектур вроде x86/ARM; на других архитектурах номера могут отличаться.

## Обработчики сигналов

У процесса есть таблица обработчиков сигналов. Для большинства сигналов процесс может:

- оставить действие по умолчанию;
- игнорировать сигнал;
- назначить свой обработчик.

Нельзя перехватить или игнорировать:

- `SIGKILL`;
- `SIGSTOP`.

Это нужно, чтобы ОС всегда могла принудительно завершить или остановить процесс.

Пример обработчика:

```c
#include <signal.h>
#include <stdio.h>
#include <unistd.h>

static volatile sig_atomic_t stop = 0;

void handle_term(int sig) {
    stop = 1;
}

int main(void) {
    signal(SIGTERM, handle_term);

    while (!stop) {
        sleep(1);
    }

    puts("graceful shutdown");
    return 0;
}
```

## Сигналы реального времени

POSIX делит сигналы на стандартные и сигналы реального времени.

Сигналы реального времени:

- находятся в диапазоне от `SIGRTMIN` до `SIGRTMAX`;
- не имеют заранее закрепленного универсального смысла;
- доставляются в порядке отправки;
- могут передавать дополнительное значение через `sigqueue`;
- часть диапазона может использовать glibc для потоков, поэтому лучше не полагаться на абсолютные номера.

## Практическое управление службами через сигналы

Типичные соглашения:

- `SIGTERM` - корректно завершиться;
- `SIGKILL` - завершить принудительно, если процесс завис;
- `SIGHUP` - перечитать конфигурацию у многих демонов;
- `SIGUSR1`/`SIGUSR2` - пользовательские действия, зависящие от программы.

Пример:

```bash
sudo kill -HUP $(pidof nginx)
sudo kill -TERM <PID>
sudo kill -KILL <PID>
```

Но для служб под `systemd` обычно лучше:

```bash
sudo systemctl reload nginx
sudo systemctl stop nginx
sudo systemctl kill nginx
```

## Что сказать на экзамене коротко

Демон - фоновый процесс без управляющего терминала, обычно работающий независимо от пользовательской сессии и обслуживающий запросы или события. Исторически демон создавали через `fork`, `setsid`, второй `fork`, закрытие дескрипторов и перенаправление потоков в `/dev/null`; сейчас чаще используют `systemd` services. Сигнал - асинхронное сообщение процессу от ядра или другого процесса. Большинство сигналов можно обработать, но `SIGKILL` и `SIGSTOP` нельзя перехватить.

## Code

```bash
# процессы и состояния
ps -eo pid,ppid,state,tty,cmd

# запустить в фоне и отвязать от shell
nohup ./worker.sh &
disown %1

# послать сигналы
kill -TERM <PID>
kill -KILL <PID>
kill -HUP <PID>

# службы systemd
systemctl status sshd
sudo systemctl restart sshd
journalctl -u sshd

# посмотреть доступные сигналы
kill -l
```
