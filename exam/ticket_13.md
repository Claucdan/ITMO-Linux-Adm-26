## Zettel

**Главная мысль:** в Linux основной способ установки ПО - пакетный менеджер дистрибутива, потому что он контролирует зависимости, обновления и удаление; ручная установка и сборка из исходников нужны для особых случаев, но хуже управляются централизованно.

## Развернутый ответ

В GNU/Linux программное обеспечение можно установить несколькими способами. Выбор зависит от дистрибутива, требований к версии, безопасности, воспроизводимости и того, нужно ли системе отслеживать установленный набор файлов.

Основные способы:

1. Из официального репозитория через пакетный менеджер.
2. Из стороннего репозитория.
3. Локальным пакетным файлом: `.deb`, `.rpm`.
4. Из исходного кода.
5. В `/opt` или `/usr/local` как вручную разложенное приложение.
6. Через универсальные форматы: Flatpak, Snap, AppImage.
7. Через language-specific менеджеры: `pip`, `npm`, `cargo`, `gem`, `go install`.
8. Через контейнеры.

## Официальные репозитории

Это предпочтительный способ для серверов и учебного администрирования.

Примеры:

```bash
sudo apt update
sudo apt install nginx
sudo dnf install nginx
sudo pacman -S nginx
```

Преимущества:

- зависимости разрешаются автоматически;
- пакеты подписаны;
- обновления приходят централизованно;
- пакетный менеджер знает, какие файлы установлены;
- удобно удалять и обновлять;
- выше воспроизводимость;
- есть интеграция с systemd, логами, пользователями, путями дистрибутива.

Недостатки:

- версия может быть старее upstream;
- пакет может отсутствовать;
- пакет собран с настройками сопровождающего дистрибутива.

Область применения:

- серверное ПО;
- системные утилиты;
- библиотеки;
- компиляторы;
- все, что должно обновляться штатно.

## Сторонние репозитории

Сторонний репозиторий добавляют, когда в официальном нет нужной версии.

Примеры:

- PPA в Ubuntu;
- vendor repository для Docker, Kubernetes, PostgreSQL;
- локальный корпоративный репозиторий;
- COPR в Fedora;
- EPEL для RHEL-подобных систем.

Плюсы:

- свежие версии;
- официальный канал от производителя программы;
- обновления через обычный пакетный менеджер.

Минусы:

- доверие переносится на владельца репозитория;
- возможны конфликты версий;
- нужно управлять ключами подписи;
- повышается риск несовместимости.

## Локальный пакетный файл

Можно скачать `.deb` или `.rpm` и установить вручную:

```bash
sudo apt install ./package.deb
sudo dpkg -i package.deb
sudo dnf install ./package.rpm
sudo rpm -i package.rpm
```

Разница:

- `apt install ./file.deb` умеет подтянуть зависимости из репозиториев;
- `dpkg -i file.deb` устанавливает пакет низкого уровня и может оставить неудовлетворенные зависимости;
- `dnf install ./file.rpm` разрешает зависимости;
- `rpm -i file.rpm` работает низкоуровнево.

Область применения:

- установка vendor-пакета;
- тест локально собранного пакета;
- пересборка пакета в лабораторной работе;
- установка без подключения репозитория.

## Установка из исходного кода

Типовой цикл:

```bash
./configure
make
sudo make install
```

Или для CMake:

```bash
cmake -S . -B build
cmake --build build
sudo cmake --install build
```

Плюсы:

- можно получить самую свежую версию;
- можно включить или отключить опции сборки;
- можно патчить исходники;
- полезно для разработки и исследования.

Минусы:

- зависимости нужно ставить отдельно;
- пакетный менеджер может не знать об установленных файлах;
- сложнее обновлять и удалять;
- выше риск сломать системные пути;
- нужна компиляционная среда.

Лучше устанавливать в:

```text
/usr/local
/opt/<app>
```

А не перезаписывать файлы дистрибутива в `/usr`.

## Ручная установка в `/opt` и `/usr/local`

Часто vendor дает архив:

```bash
tar xf app.tar.gz
sudo mv app /opt/app
sudo ln -s /opt/app/bin/app /usr/local/bin/app
```

`/opt` подходит для крупных сторонних приложений, `/usr/local` - для локально установленного ПО администратора.

Плюсы:

- просто;
- не зависит от формата пакетов;
- удобно для self-contained приложений.

Минусы:

- нет автоматических обновлений;
- нет учета файлов пакетным менеджером;
- нужно самому создавать systemd unit, пользователя, логи, PATH.

## Универсальные форматы

### AppImage

Один исполняемый файл, часто запускается без установки.

Плюсы:

- удобно для desktop-приложений;
- не требует root.

Минусы:

- слабее интеграция с системой;
- обновления зависят от приложения;
- не основной серверный способ.

### Flatpak

Ориентирован на desktop-приложения и sandbox.

Плюсы:

- изоляция;
- независимость от дистрибутива;
- удобно для графических программ.

Минусы:

- занимает больше места;
- отдельная система runtimes;
- не типичный способ для системных служб.

### Snap

Пакеты с изоляцией и автоматическими обновлениями.

Плюсы:

- единый формат;
- автоматические обновления;
- работает на разных дистрибутивах.

Минусы:

- зависимость от snapd;
- спорная модель контроля;
- не всегда подходит для минимальных серверов.

## Language-specific менеджеры

Примеры:

```bash
pip install package
npm install -g package
cargo install tool
gem install package
go install example.com/tool@latest
```

Их область:

- библиотеки и инструменты конкретного языка;
- пользовательская разработка;
- изолированные окружения.

Риск: такие менеджеры могут обходить системный пакетный менеджер. Для Python лучше использовать venv:

```bash
python3 -m venv .venv
. .venv/bin/activate
pip install flask
```

## Контейнеры

ПО можно запускать в контейнере:

```bash
docker run nginx
podman run nginx
```

Плюсы:

- изоляция зависимостей;
- воспроизводимость;
- удобно для сервисов и CI.

Минусы:

- нужен runtime;
- нужно управлять образами, volumes, сетью;
- это запуск приложения, а не установка в систему в классическом смысле.

## Что сказать на экзамене коротко

Основной способ установки ПО в Linux - пакетный менеджер дистрибутива и официальные репозитории. Это дает зависимости, подписи, обновления и учет файлов. Локальные `.deb`/`.rpm` используют для vendor-пакетов и теста. Сборка из исходников нужна для свежих версий, патчей и нестандартных опций, но хуже управляется. `/opt`, `/usr/local`, Flatpak, Snap, AppImage, language-specific менеджеры и контейнеры применяются для специальных случаев.

## Code

```bash
# Debian/Ubuntu
sudo apt update
apt show build-essential
sudo apt install build-essential
apt search nginx
apt-cache depends gcc
dpkg-query -l

# Fedora/RHEL
sudo dnf search nginx
sudo dnf install nginx
rpm -qa

# Arch
sudo pacman -Syu
sudo pacman -S nginx

# локальный пакет
sudo apt install ./app.deb
sudo dnf install ./app.rpm
```
